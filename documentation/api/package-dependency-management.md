# Package Dependency Management

We use [UV](https://docs.astral.sh/uv/) for managing our APIs dependencies.

The [pyproject.toml](../../api/pyproject.toml) file is where we configure the minimum or exact version of a dependency. These versions also get modified
by the uv commands (several described below) which can help generate / find the proper versions for a dependency.

From this configuration, a [uv.lock](../../api/uv.lock) is generated which has the specific references to dependencies to be downloaded. The lock
file should never be manually modified and should always be generated by UV.

# General Information

## Docker Base Image Updates

To limit the exposure/attack surface we build our base containers from scratch. These base images are exposed through the [GitHub Packages interface/Docker Repository](https://github.com/HHS/simpler-grants-gov/pkgs/container/python-base-image). The API and Analytics container builds/images reference those project wide Python base images as the base images to build their respective Dockerfiles.

### Locating the Vulnerability

When a vulnerability scan fails and either:
- the affects and fix versions reflects Amazon Linux and the type is `rpm`
- OR the JSON location for the python packages are in the root not the app folder `/api/.venv/`

This indicates that the vulnerabilities are in the base image and that will need to be upgrade.
Some Python packages are installed both places, so they will both flag in the JSON and upgrade in both the base image and via `pip install`

### Upgrading the Base Image
Upgrading the base image involves going to a newer (or the `latest`) tagged base image on the [Package/Container Repository page](https://github.com/HHS/simpler-grants-gov/pkgs/container/python-base-image).
1. Find the version/tag you want to upgrade to (almost always just the top one tagged latest among other things) and click it
2. Click into the "OS / Arch" tab to get the "linux/amd64" Commit SHA and validation hash, like `a586e45@sha256:22d2ff70c08fdd1ae7e029e7212a5ba29e377232995205f24dc7c64f6b6e66b0`
   <img width="726" height="405" alt="image" src="https://github.com/user-attachments/assets/1b00831b-123d-428e-9c60-7f99b4f98f80" />
3. Go to the Dockerfiles for API and Analytics and replace the first line commit hash and sha section with those values from the latest verison

4. Run `make build` in the `api` folder to ensure the build runs without any failures about the base image that was just changed.

## Dependency Versions

In our dependencies we could specify a package version in a few different ways:
* `example==1.2.0` - saying we want exactly that version and no others
* `example>=1.1.0,<1.3.0` - saying any version between 1.1.0 and before 1.3.0 would be acceptable

See [dependency-specifiers](https://docs.astral.sh/uv/concepts/projects/dependencies/#dependency-specifiers) for further details.

## Package Extras

Some dependencies have "extra" packages that can be downloaded in addition to the main package for additional optional features.
These can be specified in the `pyproject.toml` file like so: `SQLAlchemy[mypy]>=2.0.21,<3` where
each "extra" is a comma-separated value.


## Dependencies & Dev-only Dependencies

We split dependencies into dev and non-dev dependencies. Dev dependencies are ONLY installed locally and in some CI/CD commands as they are only necessary for local development (including tests, linting, formatting, and dev utilities).

Many of our dev tools are used for testing, including generating data which we want to explicitly avoid doing non-locally.
Doing this also reduces the image size, and improves security as there are fewer dependencies that could potentially run on a production server.

Dependencies are grouped under `dependencies` and `dependency-groups`. When adding a dependency,
make sure to put it in the proper group as when the API runs non-locally it will only have  packages from the first group.

* To add a dependency: `uv add boto3`
* To add a dev dependency - `uv add --dev ruff`
* See [adding-dependencies](https://docs.astral.sh/uv/concepts/projects/dependencies/#adding-dependencies)

# Common Commands

See the UV docs for more details, but here are a few useful commands for managing package dependencies in common scenarios.

## Installing / Syncing

Installing is only necessary if you're running outside of Docker (eg. you've set the `PY_RUN_APPROACH=local` env variable)

```shell
uv sync
```

This will cleanup any dependencies that are no longer present in the pyproject.toml file.

All installed dependencies can be found in the `/api/.venv` folder.

## Adding a dependency or updating minimum version

Adding/updating a dependency of at least a specific version or changing the minimum version can be done with the [add](https://docs.astral.sh/uv/concepts/projects/dependencies/) command which will update the `pyproject.toml` file for you.

```shell
# Add a dev dependency with a defined range
uv add "alembic>=1.12.0,<2"
```

Adding a dev dependency with a defined range.
```shell
uv add --dev "black>=24.0.0,<25"
```

UV does not currently have a way to upgrade a dependency to the latest in the pyproject.toml.
However, you can see what packages you are out-of-date on by running:

```shell
uv pip list --outdated
```

## Update dependencies to latest / update uv.lock

[uv lock](https://docs.astral.sh/uv/reference/cli/#uv-lock) will update the uv.lock file following the rules from the pyproject.toml file.

```shell
# Upgrade ALL dependencies based on the rules defined in the pyproject.toml
uv lock --upgrade

# can also do this to check if the lock file is up to date
uv lock --check

# Or for a specific package
uv lock --upgrade-package pydantic
```

# Verification

At a minimum, all of our formatting, linting, and tests should continue working when upgrading. If a dependency upgrade causes failures
it will need to be investigated and fixed either by adjusting the code using it (eg. a method signature in a dependency changed) or adjusting
the version we use.

Locally it is suggested you run the following commands after upgrading to verify the API still runs:
```shell
# Stop and delete DB volumes so it will be made fresh
make clean-volumes

# Rebuild
make init

# Optionally if running outside of the docker container
uv sync

# format / lint / test
make check

# Run the API and verify it locally at http://localhost:8080/docs
make run-logs
```

# Upgrading Python
Python does [yearly releases](https://devguide.python.org/versions/) for their minor versions (eg. 3.12 -> 3.13). They do not
use semvar versioning, and their [minor releases](https://devguide.python.org/developer-workflow/development-cycle/#devcycle) contain
breaking changes.

Pin to a specific minor version of python just in case anything would break when the yearly release does occur.
Only pin the minor versions of Python (eg. 3.12), and not the patch versions (eg. 3.12.1) as those contain bug and security fixes.

Along with any version upgrades, remember to:
- Test the system functionality before deploying to production
- Review the [changelog](https://docs.python.org/3/whatsnew/changelog.html)
for any breaking changes of features you may use

## Upgrade Steps
To upgrade the Python version, make changes in the following places:
1. Local Python version (see more about managing local Python versions in [development](./development.md))
2. [Dockerfile](../../api/docker-python-base)
    search for the line `ARG PYTHON_VERSION` and `ARG PY_MM` - supported versions can be found on the official python: [releases page](https://www.python.org/downloads/) The PYTHON_VERSION should include the full version number <major.minor.patch> and the PY_MM should only include the <major.minor> format.
3. Build the new python base image from [github actions](https://github.com/HHS/simpler-grants-gov/actions/workflows/cd-python-base-image.yml) and retrieve the new image tag from the [github packages](https://github.com/HHS/simpler-grants-gov/pkgs/container/python-base-image) The image tag should have the github branch name and the commit hash associated with it.
2. Update the [Dockerfile](../../api/Dockerfile) base image tag on line #1.
4. [pyproject.toml](../../api/pyproject.toml)
    search for the line
    ```toml
    [project]
    requires-python = ">=3.14,<3.15"
    ```
   Then run `uv lock` to update the [uv.lock](../../api/uv.lock) file. See [python version requirement](https://docs.astral.sh/uv/concepts/projects/config/#python-version-requirement)
5. Ensure linters and formatters are passing by running the commands below in the `api` directory.
```
make format
make lint
```
